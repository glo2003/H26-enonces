# Rétrospective sur le processus

Veuillez répondre aux questions dans votre fichier `exercices/tp2/tp2.md`.

Déjà une itération de faite! Vous avez planifié votre travail avec des issues et intégré
vos changements via des pull requests. Il est normal d’avoir eu quelques accrochages :
l’objectif de cette rétrospective est de les identifier et de trouver des pistes
d’amélioration.

## Métriques à mesurer (à inscrire dans le document d’exercice)

Pour démarrer la réflexion, mesurez les métriques suivantes pour votre itération.
Pour chacune, indiquez **moyenne / minimum / maximum** :

- Temps pour implémenter une issue
- Temps pour intégrer une pull request (**review + correctifs + merge**)
- Nombre de personnes travaillant sur chaque issue (seul, en paire, en équipe, etc.)
- Nombre de personnes qui *review* chaque pull request
- Nombre d’issues en cours d’implémentation en même temps
- Nombre de pull requests en cours de review en même temps

## Réflexions à discuter en équipe (à répondre dans le document d’exercice)

Après avoir mesuré ces métriques, discutez et répondez aux questions suivantes :

1. Selon vous, est-ce que les issues et les pull requests prenaient trop de temps à être terminées?
   Ou pas assez? Quel serait un temps idéal (approximatif) pour chacun?
2. Quel est le lien entre la **taille** des issues/pull requests et le temps nécessaire pour les terminer?
3. Donnez au moins **3** façons d’améliorer votre processus (ex. taille des issues/PR, communication,
   code review, uniformisation, gestion du WIP, etc.).
